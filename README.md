# GradConnect API

Nigeria's Graduate Career Intelligence Platform вЂ” not just another job board, but an ecosystem that helps graduates understand who is hiring, what to expect, and how to prepare.

This repository contains the Go backend API. The frontend (Next.js) lives in a separate repository.

## Tech Stack

- **Go 1.22+** (standard library `net/http` router) вЂ” modular monolith, Edwards pattern
- **PostgreSQL 16** вЂ” primary database, full-text search via `tsvector`
- **Resend / Postmark** вЂ” transactional emails
- **Google OAuth 2.0** вЂ” social login
- **golang-migrate** вЂ” database migrations
- **swaggo/swag** вЂ” OpenAPI 3.0 spec generation from handler annotations
- **Swagger UI + ReDoc** вЂ” interactive and readable API documentation

### Architecture Overview

**`cmd/api/`** вЂ” Application layer. Contains HTTP handlers, route registration, middleware, and request/response helpers. Each handler file groups related endpoints (all employer handlers in `employers.go`, all auth handlers in `auth.go`). Handlers are thin вЂ” they parse the request, call a model method, and write the response.

**`internal/data/`** вЂ” Data access layer. Each file defines a model struct (e.g. `EmployerModel`) with methods that encapsulate SQL queries. All models are collected into a `Models` struct in `models.go`, which gets initialised in `main.go` and passed to handlers via the application struct. Models don't import each other вЂ” cross-entity queries use joins within a single model method rather than calling between packages.

**`internal/mailer/`** вЂ” Email delivery. Provides a `Mailer` interface with a `Send(recipient, templateFile, data)` method and a concrete implementation for Resend or Postmark. HTML templates live in `templates/`.

**`internal/worker/`** вЂ” Background job processing. A pool of goroutines polls the `task_queue` table using `SELECT ... FOR UPDATE SKIP LOCKED`. The pool manager handles concurrency, graceful shutdown on `SIGTERM`, and retry logic with exponential backoff. Job handler functions are registered by type (e.g. `email:verify`, `email:welcome`, `session:cleanup`, `scrape:listings`).

**`internal/validator/`** вЂ” Reusable validation helpers. Provides a `Validator` struct that accumulates field errors and an `IsValid()` check. Used by handlers to validate request bodies before passing data to models.

**`migrations/`** вЂ” SQL migration files managed by `golang-migrate`. Each migration is a pair of `.up.sql` and `.down.sql` files, numbered sequentially.

**`remote/`** вЂ” Production server configuration and setup scripts. Contains systemd service files, reverse proxy config (Caddy), and provisioning scripts for the production environment.

## Getting Started

**1. Clone the repository:**

```bash
git clone https://github.com/your-username/gradconnect-api.git
cd gradconnect-api
```

**2. Set up environment variables:**

```bash
cp remote/production/.env.example .env
```

Required variables:

```
GRADCONNECT_DB_DSN=postgres://user:password@localhost:5432/gradconnect?sslmode=disable
GRADCONNECT_SMTP_HOST=smtp.resend.com
GRADCONNECT_SMTP_PORT=587
GRADCONNECT_SMTP_USERNAME=resend
GRADCONNECT_SMTP_PASSWORD=re_xxxxxxxxxxxx
GRADCONNECT_SMTP_SENDER="GradConnect <no-reply@gradconnect.ng>"
GRADCONNECT_GOOGLE_CLIENT_ID=xxxxxxxxxxxx.apps.googleusercontent.com
GRADCONNECT_GOOGLE_CLIENT_SECRET=xxxxxxxxxxxx
GRADCONNECT_CORS_TRUSTED_ORIGINS=http://localhost:3000
```

**3. Create the database:**

```bash
createdb gradconnect
```

**4. Run migrations:**

```bash
make db/migrations/up
```

**5. Seed test data:**

```bash
make db/seed
```

**6. Generate API docs:**

```bash
make docs/generate
```

**7. Start the API server:**

```bash
make run/api
```

The server starts on `:4000` by default. Visit `http://localhost:4000/api/v1/docs/swagger/` for the interactive API explorer.

## API Documentation

The API is a RESTful JSON service prefixed at `/api/v1`. Documentation is available in three forms:

| Format | URL | Description |
|--------|-----|-------------|
| Swagger UI | `/api/v1/docs/swagger/` | Interactive API explorer вЂ” test endpoints directly (development only) |
| ReDoc | `/api/v1/docs/redoc` | Clean, readable reference documentation (available in all environments) |
| Markdown | `GradConnect_API_Specification_v1_0.md` | Full endpoint contracts with request/response examples |

Swagger and ReDoc are auto-generated from handler annotations using `swaggo/swag`. Run `make docs/generate` after changing any handler annotations.

**Public endpoints** (no auth required): employer profiles, opportunity listings, assessment guides, community reviews.

**Authenticated endpoints** (session cookie required): bookmarks, application tracker, review submission, profile management.

**Verification-gated endpoints**: write operations (bookmark, track, review) require a verified email. Unverified users get full read access. The middleware returns `403 { "error": { "code": "email_verification_required" } }` for gated actions, which the frontend intercepts to show a verification prompt.

## Database

14 tables in the MVP schema. Full column specs, constraints, indexes, and enum types are in `GradConnect_Database_Design_v1_0.md`.

Core entities: `employer`, `opportunity`, `assessment_profile`, `app_user`, `review`.

Auth/session: `session`, `password_reset_token`, `email_verification_token`.

User actions: `bookmark`, `application_track`, `user_permission`.

Infrastructure: `task_queue`.

Editable entities (`employer`, `assessment_profile`, `app_user`) use optimistic concurrency control via a `version` column.

## Background Jobs

The worker pool starts alongside the HTTP server in `main.go`. It consumes tasks from the `task_queue` table with configurable concurrency (default: 5 goroutines).

| Job Type | Trigger |
|----------|---------|
| `email:verify` | User registration (email auth) |
| `email:welcome` | After verification or Google OAuth registration |
| `email:password_reset` | Forgot password request |
| `email:deadline_reminder` | Daily cron вЂ” verified users with bookmarks expiring within 3 days |
| `session:cleanup` | Hourly cron вЂ” delete expired sessions |
| `scrape:listings` | Scheduled вЂ” scrape and deduplicate opportunity listings |

Failed tasks retry with exponential backoff (1 min в†’ 5 min в†’ 30 min). After 3 attempts, tasks are marked `dead` for manual inspection.
